00 / Product goal

Що ми будуємо і навіщо

Це навчальний MVP персонального Q&A бота. Його головна цінність не в кількості фіч, а в тому, що весь RAG шлях видно в коді: від локального файлу до відповіді з джерелами.

Learn побачити базові LangChain building blocks на маленькому реальному прикладі.
Trust відповідати тільки з документів і показувати sources після відповіді.
Reuse не перебудовувати Chroma index, якщо файли у docs/ не змінилися.
Ship мати локальний запуск, CI checks і простий Render deploy через Blueprint.

Поточний принцип продукту: спочатку зрозумілий вертикальний slice, потім покращення якості. Тому в MVP intentionally простий local Chroma, простий React UI і один спільний QA service для CLI та API.

02 / Ingestion

Як документи потрапляють у Chroma

Ingestion - це підготовка знань для пошуку. Тут сирі файли стають маленькими chunks із metadata, embeddings і стабільними chunk ids.

1. Read files load_text_files(DOCS_DIR) читає тільки .md і .txt.
2. Metadata source, file name, type, length, checksum, modified time.
3. Plan manifest порівнюється з поточними файлами: added, changed, deleted.
4. Split RecursiveCharacterTextSplitter: 300 chars, 60 overlap.
5. Embed OpenAIEmbeddings створює vectors для кожного chunk.
6. Persist Chroma index і ingestion_manifest.json зберігаються локально.

Коли index перевикористовується

Якщо data/index/chroma/ існує, manifest існує, і plan каже needs_rebuild = false, код відкриває готовий Chroma index замість нового rebuild.

Коли index перебудовується

Якщо файл додали, змінили або видалили, система заново ріже всі документи на chunks, перераховує embeddings і записує новий manifest. Це простіше, ніж partial update, і добре для MVP.

Що тут робить LangChain

Document тримає текст і metadata, RecursiveCharacterTextSplitter робить chunks, а OpenAIEmbeddings перетворює текст у vectors для semantic search.

Що тут робить наш код

Наш код вирішує, які файли брати, які metadata зберігати, як називати chunk_id, коли rebuild потрібен, і де саме лежать generated files.

03 / Question answering

Шлях одного питання через RAG

І CLI, і API зрештою приходять у answer_question(). Тому важливо розуміти саме service layer у qa_service.py.

1

Validate input

validate_question() прибирає пробіли і відкидає порожнє питання.

2

Normalize chat history

Беруться тільки ролі user/assistant, максимум 6 останніх змістовних повідомлень.

3

Rewrite follow-up question

Якщо є історія, модель переписує питання в standalone form, наприклад “А чим вона відрізняється від ND?” → “Чим MX-5 NA відрізняється від ND?”.

4

Retrieve context

Retriever шукає top-k chunks у Chroma. get_search_k() не просить більше chunks, ніж реально є в collection.

5

Format context and sources

format_context() збирає fragments для prompt, а build_sources() готує список джерел для UI/API.

6

Generate answer

Prompt змушує відповідати українською тільки з retrieved context. Якщо контекст не містить відповіді, очікувана фраза: Я не знайшов цього в документах.

Важлива деталь для follow-up питань: retrieval шукає не оригінальне коротке питання, а standalone_question. Це допомагає, коли користувач питає “А чим вона відрізняється?” після попередньої репліки про конкретну модель авто.

04 / API and frontend

FastAPI тонкий, React показує stream

HTTP layer майже не містить бізнес-логіки. Він приймає запит, викликає service layer і повертає або JSON, або Server-Sent Events для поступового виводу відповіді.

POST /ask

Синхронний endpoint. Повертає повну відповідь і sources одним JSON response.

POST /ask/stream

Streaming endpoint. Спочатку надсилає sources, потім answer_chunk, в кінці done.

React chat

App.jsx читає SSE stream через response.body.getReader() і дописує текст бота chunk за chunk.

/ask JSON

{
  "question": "Яка різниця між MX-5 NA і ND?",
  "chat_history": []
}

Зручно для Swagger UI, простих тестів і CLI-like інтеграцій, де відповідь можна чекати одним response.

/ask/stream SSE

event: sources
event: answer_chunk
event: error
event: done

Зручно для chat UI, бо користувач бачить відповідь поступово, навіть якщо model generation займає кілька секунд.

Важлива межа: frontend не знає LangChain. Він лише збирає question і chat_history, показує повідомлення та sources. Уся RAG-логіка залишається в backend.

05 / LangChain checkpoints

Що означають головні RAG поняття у цьому коді

LangChain тут не магія, а набір маленьких деталей, які ми з'єднали в pipeline. Нижче - як кожен термін проявляється саме в цьому репозиторії.

D

Document

LangChain Document має текст у page_content і службову інформацію в metadata.

C

Chunk

Малий фрагмент документа. Overlap допомагає не втрачати сенс на межі двох шматків.

V

Vector store

Chroma зберігає embeddings і дозволяє знайти фрагменти, схожі на питання.

P

Prompt boundary

Фінальна відповідь має опиратися тільки на context. Це головний захист від вигаданих фактів.

Поняття Де в коді Навіщо потрібно
LCEL chain build_retrieval_chain(), build_generation_chain() З'єднує кроки: input, retrieval, context formatting, prompt, model, parser.
Retriever build_retriever() Дає однаковий interface для пошуку chunks у Chroma.
ChatPromptTemplate build_answer_prompt(), build_question_rewrite_prompt() Тримає правила: відповідати українською, не вигадувати, переписувати follow-up.
StrOutputParser qa_service.py Перетворює відповідь chat model у звичайний рядок для API або CLI.
Source preview build_sources() Повертає користувачу не лише answer, а й джерела, з яких вона була зібрана.

06 / Runtime and deploy

Як проєкт живе локально, у CI та на Render

Локально ми запускаємо backend і frontend окремо. У CI перевіряємо, що зміна не ламає код. На Render Blueprint створює два сервіси: Python API і static React frontend.

Local dev dev
  1. make api запускає FastAPI на 127.0.0.1:8000.
  2. make frontend-dev запускає Vite на 127.0.0.1:5173.
  3. frontend/.env задає VITE_API_BASE_URL.
CI checks CI
  1. backend-tests запускає make test.
  2. frontend-build запускає production build.
  3. secret-scan шукає випадково додані ключі.
Render backend API
  1. Build ставить backend/requirements.txt.
  2. Start command спочатку запускає indexing.
  3. Потім стартує uvicorn і відкриває /health.
Render frontend web
  1. Build виконує npm ci і npm run build.
  2. Static site публікує frontend/dist.
  3. Rewrite route повертає index.html для будь-якого path.

На Render free plan filesystem може бути ephemeral, тому data/index/ не вважаємо durable database. Для demo backend може перебудувати index під час startup, а в майбутньому можна додати persistent disk або винести vector store окремо.

07 / What we want to reach

Куди рухається MVP

Поточна версія вже показує повний RAG цикл. Наступні покращення мають робити систему надійнішою, зрозумілішою для навчання і ближчою до реального персонального помічника.

Quality Кращий retrieval

Підібрати chunk size, overlap, k, eval cases і перевірити comparison/follow-up сценарії.

UX Більше прозорості

Показувати source previews, loading states, errors і корисні пояснення без перевантаження UI.

Data Зручніше оновлення docs

Додати зрозумілий flow для додавання документів і перевірки, що index перебудувався правильно.

Ops Стабільніший production

Розглянути persistent storage для data/index/, кращі smoke tests і production observability.

Definition of done для навчального кроку

  • Новачок може пояснити, що таке chunk, embedding, vector store, retriever і prompt.
  • Новачок може запустити backend tests без OpenAI key.
  • Новачок розуміє, які команди вже викликають OpenAI і потребують real key.

Definition of done для продуктового кроку

  • Відповідь має sources і не вигадує факти поза документами.
  • Follow-up питання працює через chat history rewrite.
  • Frontend, API, CI і Render config лишаються узгодженими.

08 / Tests and eval

Що перевіряється автоматично

Unit tests тут спеціально мережево-незалежні: вони мокають OpenAI і перевіряють маленькі функції. RAG eval окремий, бо він уже залежить від реального індексу, моделі та мережі.

Шар Що покриває Команда
Loader tests читання тільки .md/.txt, metadata, preview make test
RAG helper tests chunk metadata, manifest, ingestion plan, prompts, chat history make test
API tests /ask, /ask/stream, CORS, SSE events make test
RAG eval expected sources, must-include phrases, fallback answer make eval

09 / Practical commands

Команди, які найчастіше потрібні

Backend

make install
make test
make load-docs
make index-demo
make ask QUESTION="Яка різниця між MX-5 NA і ND?"
make api

Frontend

make frontend-install
make frontend-dev
make frontend-build

make test не потребує OpenAI API. А make index-demo, make ask, make eval і make api потребують коректний OPENAI_API_KEY у кореневому .env.

Оновлено як навчальна візуальна карта поточного стану репозиторію. Згенеровані індекси, .env, .venv, node_modules і build artifacts не мають потрапляти в git.
